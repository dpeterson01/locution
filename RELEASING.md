# Releasing Locution

Reference for cutting a signed release. Locution ships an auto-updater, so every
release must be signed with the updater key or existing installs will reject it.

## Updater signing key

Releases are signed with a minisign keypair generated by the Tauri signer.

- **Private key (secret, never commit):** `~/.tauri/locution.key`
- **Passphrase (secret, never commit):** chosen when the key was generated
- **Public key (committed):** `src-tauri/tauri.conf.json` → `plugins.updater.pubkey`

The private key and passphrase live only on the release machine and in GitHub
Actions secrets. They are never stored in the repo.

### Required GitHub Actions secrets

Set these in the repo settings (Settings → Secrets and variables → Actions):

| Secret                               | Value                                         |
| ------------------------------------ | --------------------------------------------- |
| `TAURI_SIGNING_PRIVATE_KEY`          | Full contents of `~/.tauri/locution.key`      |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | The passphrase set when the key was generated |

### Regenerating the key (only if lost or rotating)

```bash
bunx tauri signer generate -w ~/.tauri/locution.key
```

This prints a **public** key. Rotating the key is a breaking change for the
updater: copy the new public key into `plugins.updater.pubkey`, update both
GitHub secrets, and ship a release. Installs on the old key can no longer
auto-update and must be reinstalled once, so only rotate if the private key is
compromised or lost.

## Cutting a release

1. Bump the version in **both** places (keep them in sync):
   - `package.json` → `version`
   - `src-tauri/tauri.conf.json` → `version`
2. Update the release notes under `src/content/release-notes/`.
3. Commit, tag, and push. The release workflow (`.github/workflows/release.yml`)
   runs with `sign-updater: true` so it produces signed updater artifacts and a
   `latest.json` manifest attached to the GitHub release.
4. The updater endpoint is
   `https://github.com/dpeterson01/locution/releases/latest/download/latest.json`.
   Existing installs check it and offer the update once the release is published.

## Signing notes

- `sign-updater` (updater artifact signing) is decoupled from `sign-binaries`
  (OS code signing). Releases sign the updater artifacts but leave
  `sign-binaries: false`; OS-level code signing (Apple/Azure) is a separate,
  currently-unused path and does **not** need the updater secrets.
- Only the two secrets above are required to publish a working auto-update.
